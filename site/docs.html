The tracker has two linked layers. The first layer is a legal event registry for tariff actions. The second layer is a curated case library of empirical treatment-control designs. The event is the legal unit. The case is the empirical unit. One event can have multiple cases.

How to read the tabs

Feed: the main working view. Select an event, inspect attached cases, and view the chart and summary metrics.

Map: geography-first navigation for tracked tariff events.

Event Registry: the legal event layer with filters for authority, status, and case coverage.

Case Library: the research-facing view of live empirical cases by event and stage.

Build Queue: the operational queue for important events that still need case-building.

Worldwide: a separate reporter-country trade-score workspace built from bilateral rows. It does not replace the legal event / empirical case architecture.

How the worldwide free-trade score is calculated

The worldwide layer uses a bilateral goods-trade score for each reporter-partner pair, then rolls those pair scores up into a reporter-country summary. The current live method is a simple 0–100 friction score where higher means lower trade friction.

Pair-level score formula:
goods_score_live_v1 = clamp(100 - 4 × effective applied tariff % - NTM penalty points - trade-remedy penalty points)

Tariff input used in the pair score:
The score uses the pair’s effective applied tariff field. In the live build, that field comes first from WITS bilateral weighted applied tariff data when available. If that bilateral tariff is missing, the scorer falls back to the reporter country’s trade-weighted MFN tariff. If bilateral preference overrides are later applied for a specific reporter-partner pair, that pair’s tariff input is replaced with the curated bilateral preferential tariff and the score is recomputed.

Component meaning:
The score starts at 100. Tariffs reduce the score at a rate of 4 points for each 1 percentage point of effective applied tariff. Non-tariff-measure penalty points reduce the score directly. Trade-remedy penalty points also reduce the score directly. The result is then clamped so it stays within a 0–100 range.

Country summary score:
A reporter country’s headline score on the Worldwide page is the import-weighted average of its bilateral partner scores, using bilateral import value in U.S. dollars as the weight. That means larger trading relationships matter more than smaller ones.

Map shading:
The globe is colored from the reporter-country weighted_goods_score, not directly from any single tariff field. Countries with no reporter summary use a neutral fill, and countries still in the acquisition queue use a separate in-progress fill.

Other country-level fields:
Weighted tariff baseline is the import-weighted average of bilateral effective applied tariff values. Agreement trade share is the share of covered imports coming from partners marked as having an agreement in force. Best partner and worst partner are the highest-scoring and lowest-scoring partner rows for that reporter in the current data.

The score is useful as a comparative trade-friction indicator inside this project. It is not a comprehensive legal measure of all barriers, and it should not be read as a full welfare estimate or a complete bilateral market-access index.

Core methods

Case studies use curated public price-series treatment and control pairs. Treatment and control are rebased to a common base month. The reported relative effect is treatment minus control at standard post-event horizons.

The tracker is curated, not exhaustive. A live case appears only when there is a defensible public-series design attached to a specific legal event.

Sources

Legal-event pages and registry entries use official legal or administrative sources where available. Case designs use public price series and curated metadata. The worldwide layer uses bilateral country-pair trade-score inputs exported into site data.

Workflow and source of truth

The repo is spec-first. Case specs are the source of truth for case metadata. Live site/data is a build artifact, not an authoring layer. Local verification comes before push.
